# Role Alignment: Inside Product Solution Architect

## Role Fit Thesis

An Inside Product Solution Architect needs to connect customer problems, technical solutions, business value, and sales execution. This project demonstrates that full chain.

## Capability Mapping

| Role Capability | Evidence From Project |
| --- | --- |
| Understand customer business pain | Frames cloud cost pressure as a TCO and workload placement problem |
| Translate pain into solution criteria | Converts cost categories into cloud/local comparison inputs |
| Communicate financial impact | Shows TCO, savings, savings percent, run-rate delta, and break-even |
| Explain technical tradeoffs | Uses hybrid recommendation instead of binary cloud vs on-prem thinking |
| Support sales conversations | Includes one-page summary, demo script, report notes, and CSV export |
| Build trust with stakeholders | Keeps assumptions and formulas visible |
| Handle ambiguity | Uses confidence levels and validation language |
| Work cross-functionally | Project artifacts support technical, financial, and sales audiences |

## How To Explain The Role Connection

Use this answer:

```text
The role is not only about knowing infrastructure products. It is about helping customers make better decisions. I built this project to show that I can take a customer concern, model the business impact, explain the technical options, and guide the next step in a way that sales, finance, and IT stakeholders can all understand.
```

## Presales Behaviors Demonstrated

### Discovery

The dashboard input categories mirror discovery:

- Where is your cloud spend concentrated?
- How fast is the workload growing?
- What discounts already exist?
- What would local infrastructure cost to operate?
- What one-time migration costs would affect payback?

### Qualification

The recommendation qualifies whether the opportunity is:

- Strong enough for repatriation discussion.
- Better positioned as hybrid.
- Not economically justified.
- Too close to call without better assumptions.

### Value Framing

The app explains value in business terms:

- Total cost.
- Savings.
- Payback timing.
- Cost composition.
- Risk of late break-even.

### Technical Credibility

The implementation separates:

- UI.
- Cost models.
- TCO calculator.
- Recommendation rules.
- Export logic.
- Tests.

That makes the project defensible in a technical review.

## Interview Positioning

Avoid saying:

```text
I built a Streamlit app.
```

Say:

```text
I built a presales decision-support tool that uses a technical cost model to guide customer conversations about hybrid workload placement.
```

## What This Says About Me

This project is designed to show that I can:

- Think commercially.
- Communicate clearly.
- Build working technical assets.
- Understand infrastructure economics.
- Avoid overselling.
- Align technical recommendations with customer outcomes.
